很多 AI 应用一开始都能演示，但一进入真实业务就会暴露问题：上下文塞得太满、工具权限太宽、错误无法复现、评测只靠人工感觉。上下文工程要解决的不是“提示词写得更玄”，而是把模型周围的输入输出变成可维护接口。

![AI Agent 上下文边界示意图](/images/ai-agent-boundary.svg)

## 上下文是一份契约

一个可维护的 Agent 应该明确五类输入：用户意图、系统约束、检索材料、工具结果和历史状态。每一类输入都要标明来源、时效、可信度和可见范围。

如果所有内容都被拼成一大段文本，模型可能短期看起来“更懂”，长期却会变得难调试。更好的方式是把上下文拆成结构化片段：

```json
{
  "intent": "生成变更风险摘要",
  "sources": ["incident-log", "deployment-diff"],
  "tools_allowed": ["read_logs", "read_metrics"],
  "must_not_do": ["write_ticket", "deploy_change"]
}
```

这不是为了让模型少做事，而是让模型知道当前任务边界。

## 工具调用要有最小权限

工具是 Agent 的能力放大器，也是风险放大器。读取日志、查询知识库、创建工单、修改配置和触发发布，不应该被放在同一个权限层里。

工程上可以把工具分为只读、建议、写入和高风险四类。默认只开放只读工具；建议类工具可以生成草稿；写入类工具需要审计；高风险工具需要人工确认和明确回滚路径。

## 评测要覆盖失败方式

AI 应用的评测不应该只问“回答是否好”，还要问“什么时候会坏”。至少需要覆盖四类样本：正常任务、缺失信息、冲突信息和越权请求。

每次改提示词、检索策略或工具 schema，都应该跑同一批评测样本。这样才能知道改动是普遍变好，还是只是让某个演示案例看起来更漂亮。

## 日志与审计

上线后的 Agent 要记录输入摘要、检索来源、工具调用、模型输出和人工改写。日志不必保存所有敏感原文，但必须能回答三个问题：它为什么这么回答，调用了哪些工具，谁批准了有副作用的动作。

AI 工程的核心不是追求“更像人”，而是让系统在不确定性中仍然可解释、可回滚、可评测。上下文工程就是把这件事落到接口、权限和验证流程里。
