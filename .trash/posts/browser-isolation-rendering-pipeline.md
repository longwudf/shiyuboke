浏览器不是一个单线程页面容器，而是一组带边界的进程、线程、队列和缓存。前端性能问题、跨站攻击面和“为什么这个页面卡”往往都能在这套结构里找到线索。

![浏览器进程隔离示意图](/images/browser-isolation-diagram.svg)

## 站点隔离先于渲染优化

现代浏览器会尽量把不同站点放进不同 renderer process。这样做会增加一定进程开销，但它换来的是更清晰的安全边界：某个页面被利用时，攻击者更难直接读取另一个站点的内存。

这也解释了为什么前端排查不能只看 JavaScript。跨站 iframe、第三方脚本、Service Worker、COOP/COEP、Cookie SameSite 和 CSP 都会改变页面的隔离关系。安全配置越混乱，浏览器为了兼容越容易进入复杂路径。

## 渲染流水线的四个阶段

第一阶段是网络和缓存。DNS、TLS、HTTP 缓存、Service Worker 和预加载策略决定资源什么时候进入解析器。很多“首屏慢”问题不是 React 或 Vue 的锅，而是关键 CSS、字体和脚本阻塞了资源队列。

第二阶段是解析。HTML 生成 DOM，CSS 生成 CSSOM，脚本可能暂停解析并修改结构。这里的关键判断是：哪些脚本必须同步执行，哪些可以延迟，哪些应该拆到交互后。

第三阶段是布局。布局阶段要计算盒模型、文字换行和几何位置。频繁读取布局信息再写样式，会触发 forced reflow，让渲染主线程不断回头重算。

第四阶段是绘制与合成。浏览器会把内容绘制到图层，再交给合成线程组合。动画如果只改 `transform` 和 `opacity`，通常更容易落在合成路径；如果不断改宽高、阴影和布局相关属性，就更容易拖慢主线程。

## 安全配置也会影响体验

安全头不是孤立的合规项。CSP 可以减少第三方脚本风险，但过宽的 `script-src` 仍然会让供应链风险进入页面。COOP/COEP 可以帮助页面获得更强隔离能力，但会暴露跨源资源配置不一致的问题。

工程上可以按以下顺序推进：

1. 先梳理第三方脚本和 iframe，确认它们是否真的需要出现在首屏。
2. 再配置 CSP 报告模式，观察真实违规项。
3. 然后把关键路径资源拆出来，降低解析和布局阶段阻塞。
4. 最后用性能面板确认瓶颈到底在网络、脚本、布局还是合成。

浏览器的安全边界和性能边界常常是一回事。理解隔离与渲染流水线，才能把“页面慢”和“页面危险”放进同一套工程语言里讨论。
